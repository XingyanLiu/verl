> **⚠️ 本文档已废弃**
>
> 本文档基于错误前提：v0.7 引入了 K 倍 loss 缩放差异，需要调整超参数来补偿。
>
> **经仔细审查源码后发现：v0.3 和 v0.7 在 dp_actor 路径下的有效梯度完全相同。**
> 两个版本都除以 K = ppo_mini_batch_size / ppo_micro_batch_size_per_gpu。
> **不需要为"版本间 loss 缩放差异"调整 grad_clip、lr 或其他超参数。**
>
> 如果用户在 v0.7 上训练效果不如 v0.3，原因可能来自**算法变更**（dual-clip、ratio clamping 等），
> 而非 loss 缩放。请参考 `04_解决方案建议.md`（修正版）。

## 一、迁移前的影响评估

### 1.1 你需要关心的改动

| 改动 | 影响 | 是否需要调参 |
|------|------|-------------|
| loss 全局归一化 | pg_loss 值缩小 K 倍 | 可能需要 |
| 梯度裁剪行为 | 裁剪频率降低 | **是** |
| 多 loss 相对权重 | 不变（pg_loss 和 entropy_loss 同步缩放） | 否 |
| Dual-clip | 新增 clip_ratio_c | 注意默认值 |
| Log-ratio clamp | 新增稳定性 clamp | 通常不需调整 |

### 1.2 确定你的缩放因子 K

```
K = gradient_accumulation_steps

其中（源码验证见 `07_缩放因子K的精确推导.md`）：
gradient_accumulation_steps = ppo_mini_batch_size / ppo_micro_batch_size_per_gpu
```

- `ppo_mini_batch_size` 是**全局**值（across all workers）
- **K 与 dp_size（GPU 数量）无关**，因为 FSDP 梯度平均在新旧版本中被同等应用

例如：
- `ppo_mini_batch_size = 128`, `ppo_micro_batch_size_per_gpu = 4` → K = 32
- `ppo_mini_batch_size = 64`, `ppo_micro_batch_size_per_gpu = 8` → K = 8

## 二、迁移方案

### 方案 A：最小改动迁移（推荐）

**原则**：只调整梯度裁剪阈值，其他保持不变。

**依据**：AdamW 的自适应步长对均匀的 loss 缩放几乎不变（见 05 号文档分析），
所以学习率不需要调整。但梯度裁剪阈值需要同步缩小。

```yaml
actor_rollout_ref:
  actor:
    # ===== 关键调整 =====
    # 旧版 grad_clip = 1.0，新版需要缩小 K 倍
    grad_clip: 0.03125   # = 1.0 / 32（如果 K=32）

    # 其他参数保持不变
    loss_agg_mode: "token-mean"
    clip_ratio: 0.2
    clip_ratio_low: 0.2
    clip_ratio_high: 0.2      # DAPO: 0.28
    entropy_coeff: 0.01        # 不需要调整！
```

**验证方法**：
1. 对比旧版本和新版本的 `actor/grad_norm`（裁剪前的梯度范数）
2. 新版本的 grad_norm 应约为旧版本的 1/K
3. 裁剪后的有效梯度应与旧版本相近

### 方案 B：调整学习率迁移

**原则**：如果不想调梯度裁剪，可以反向调整学习率来补偿。

```yaml
actor_rollout_ref:
  actor:
    optim:
      # 旧版 lr = 1e-6，新版放大 K 倍
      lr: 3.2e-5       # = 1e-6 × 32
      grad_clip: 1.0    # 保持不变
```

**注意**：这种方案会改变梯度裁剪的行为：
- 学习率放大 → optimizer step 更大
- 但梯度裁剪仍然在缩小的梯度上操作
- 可能需要同时调整 grad_clip

**不推荐单独使用，除非你确认旧版本训练从未触发梯度裁剪。**

验证方法：检查旧版本训练日志中 `actor/grad_norm` 是否经常超过 `grad_clip`（默认 1.0）。
如果 `grad_norm > 1.0` 的比例超过 10%，说明梯度裁剪在旧版本中起了重要作用，
迁移时必须调整 `grad_clip`。

### 方案 C：切换 loss_agg_mode

**原则**：使用不受全局 batch size 影响的聚合模式。

```yaml
actor_rollout_ref:
  actor:
    loss_agg_mode: "seq-mean-token-mean"
    # 或
    loss_agg_mode: "seq-mean-token-sum-norm"
    loss_scale_factor: 1024     # 设为固定值
```

**特点**：
- `seq-mean-token-mean`：每个序列先做 token-mean，再做 seq-mean
  - loss 量级与旧版本更接近
  - 但语义上与旧版 token-mean 不同（长短序列权重相同）
- `seq-mean-token-sum-norm`：除以固定的 `loss_scale_factor`
  - 可以精确控制 loss 量级

**风险**：切换 loss_agg_mode 相当于改变了 loss 函数本身，不是纯粹的 scale 变化。

### 方案 D：综合方案（最稳健）

```yaml
actor_rollout_ref:
  actor:
    # 1. 保持 token-mean 模式
    loss_agg_mode: "token-mean"

    # 2. 同比缩小 grad_clip
    grad_clip: 0.03125     # = old_grad_clip / K

    # 3. 保持学习率不变
    optim:
      lr: 1e-6             # 不变

    # 4. entropy_coeff 不需要调整
    entropy_coeff: 0.01    # 不变

    # 5. 新版本特有参数（如使用 DAPO）
    clip_ratio_low: 0.2
    clip_ratio_high: 0.28
    clip_ratio_c: 3.0
```

## 三、逐步验证流程

### Step 1：对比基准建立

用旧版本 (v0.3.x) 运行 **50 步**，记录：
- `actor/pg_loss` 的量级和趋势
- `actor/grad_norm` 的范围
- `actor/pg_clipfrac` 的值
- `actor/ppo_kl` 的值
- reward 曲线

### Step 2：新版本基线运行

用新版本 (v0.7+) 使用**相同配置**运行 50 步，记录同样的指标。

对比：
```
pg_loss_new ≈ pg_loss_old / K ?          → 确认 K 值
grad_norm_new ≈ grad_norm_old / K ?      → 确认梯度缩放
pg_clipfrac 和 ppo_kl 是否相近 ?          → 确认策略更新行为
```

### Step 3：应用迁移方案

根据 Step 2 的结果选择迁移方案（推荐方案 A 或 D），运行 50 步。

验证：
```
裁剪后 grad_norm ≈ 旧版本的裁剪后 grad_norm ?  → 梯度裁剪行为一致
reward 趋势与旧版本相似 ?                        → 训练效果一致
```

### Step 4：全量训练

如果 Step 3 的指标正常，开始全量训练。

持续监控：
- reward / accuracy 曲线
- `actor/ppo_kl`：应保持在合理范围（0.001 ~ 0.1）
- `actor/pg_clipfrac`：应在 0.1 ~ 0.5（过高说明更新太激进）
- `actor/grad_norm`：应稳定，无异常尖峰

## 四、常见陷阱

### 4.1 不要只调学习率

**错误做法**：
```yaml
lr: 3.2e-5    # 放大 32 倍
grad_clip: 1.0  # 不变
```

**问题**：梯度裁剪在缩小的梯度上操作，等效于梯度裁剪阈值放大了 32 倍。
学习率放大 32 倍 + 裁剪保护减弱 → 训练不稳定。

### 4.2 不要忽略梯度裁剪

如果你的旧版训练**经常触发梯度裁剪**（`grad_norm > grad_clip` 比例 > 50%），
则梯度裁剪是训练稳定性的重要组成部分。
新版本中必须同步调整 `grad_clip`。

### 4.3 entropy_coeff 不需要调整

两个 loss 使用相同的 `agg_loss` 归一化，相对权重不变。
不要因为看到 pg_loss 变小就调整 entropy_coeff。

### 4.4 注意 dual-clip 的影响

新版本引入的 `clip_ratio_c`（默认 3.0）会影响负 advantage 的处理。
如果旧版本没有 dual-clip，需要确认这个新特性是否符合预期。

可以设置 `clip_ratio_c` 为很大的值（如 100.0）来禁用 dual-clip：
```yaml
clip_ratio_c: 100.0  # 实质上禁用 dual-clip
```

### 4.5 关注 FP16 精度

如果使用 FP16（而非默认 BF16），loss 缩小 32 倍可能导致：
- 梯度更接近 FP16 的下溢边界
- GradScaler 需要更频繁地调整 scale
- 建议切换到 BF16 或调整 GradScaler 的 `growth_interval`

## 五、快速迁移 Checklist

```
□ 1. 确认你的 K 值（K = ppo_mini_batch_size / ppo_micro_batch_size_per_gpu）
□ 2. 设置 grad_clip = old_grad_clip / K
□ 3. 保持 lr 不变
□ 4. 保持 entropy_coeff 不变
□ 5. 确认 clip_ratio_low/high 设置（DAPO: 0.2/0.28）
□ 6. 确认 clip_ratio_c 设置（如不需要 dual-clip，设为大值）
□ 7. 运行短训练验证 grad_norm、pg_clipfrac、ppo_kl
□ 8. 对比 reward 趋势
□ 9. 全量训练
```
