# verl 版本升级后 pg_loss 幅度变小的问题分析

## 问题描述

使用 verl-0.7.0 跑 DAPO RL 训练任务（之前 verl-0.3.x 可正常训练），发现相同设定下 `pg_loss` 的幅度小了一到两个数量级，导致模型优化不动。

## 核心结论

**pg_loss 幅度变小的根本原因是 loss 归一化方式的重大变更：**

| 维度 | v0.3.x | v0.7.0+ (当前) |
|------|--------|----------------|
| 归一化方式 | `masked_mean`: 在 **micro-batch** 内做 token-mean | `agg_loss(token-mean)`: 用 **global batch** 的 token 总数做归一化 |
| 分母大小 | micro-batch 内有效 token 数（较小） | 全局 batch 的有效 token 总数（很大） |
| pg_loss 量级 | 正常量级 | 缩小约 `(gradient_accumulation_steps × dp_size)` 倍 |
| 梯度正确性 | 正确（但对并行度敏感） | 正确且"并行度无关"（parallelism-agnostic） |

**这不是一个 bug，而是一个 intentional design change（有意的设计变更）**，目的是让 loss 值在不同数据并行度下保持一致。但这会导致：

1. **报告的 pg_loss 数值显著变小**（因为分母从 micro-batch tokens 变为 global batch tokens）
2. **如果学习率等超参是基于旧版本的 loss 量级调优的，直接迁移会导致训练不充分**
3. **需要相应调整学习率或 loss_scale_factor 来补偿**

## 关键影响因素

- 若使用 4 个 GPU（dp_size=4），8 个 gradient accumulation steps，则 pg_loss 约变小 **32 倍**
- 若使用 8 个 GPU（dp_size=8），4 个 gradient accumulation steps，则 pg_loss 约变小 **32 倍**
- 更大的训练规模 → pg_loss 缩小的倍数更大

## 快速验证方法

将 `micro_batch_size_per_gpu` 设为 1，可以让每个 micro-batch 只有一个样本，此时 pg_loss 值与 v0.3.x 的 per-sequence mean 近似一致。
