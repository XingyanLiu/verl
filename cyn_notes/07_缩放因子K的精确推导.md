# 缩放因子 K 的源码验证（修正版）

> **⚠️ 重要修正**：
> 之前的分析错误地声称 v0.7 引入了新的 loss 缩放，导致与 v0.3 存在 K 倍梯度差异。
> 经仔细审查后发现：**v0.3 和 v0.7 都以相同方式除以 gradient_accumulation，
> 两个版本在 dp_actor 路径下的有效梯度完全相同，不存在版本间的 K 倍差异。**

---

## 一、核心结论

| 项目 | 结论 |
|------|------|
| v0.3 和 v0.7 的 loss 缩放 | **完全相同**：都除以 K = ppo_mini_batch_size / ppo_micro_batch_size_per_gpu |
| 版本间梯度差异 | **不存在** K 倍差异 |
| K 是否包含 dp_size | **不包含** |
| 实际差异来源 | 算法变更（dual-clip、ratio clamping）和 metric 报告格式 |

---

## 二、源码对比：v0.3 与 v0.7 的 loss 除以 gradient_accumulation

### 2.1 v0.3.x（`dp_actor.py` 第 295, 347 行）

```python
# v0.3 dp_actor.py
self.gradient_accumulation = self.config.ppo_mini_batch_size // self.config.ppo_micro_batch_size_per_gpu

# ... 在 micro-batch 循环中：
pg_loss = core_algos.compute_policy_loss(...)  # masked_mean 聚合
policy_loss = pg_loss - entropy_loss * entropy_coeff
loss = policy_loss / self.gradient_accumulation    # ← 除以 K
loss.backward()
```

### 2.2 v0.7+（`dp_actor.py` 第 562-563, 585, 663 行）

```python
# v0.7 dp_actor.py
self.gradient_accumulation = (
    self.config.ppo_mini_batch_size // self.config.ppo_micro_batch_size_per_gpu
)

# ... 在 micro-batch 循环中：
loss_scale_factor = 1 / self.gradient_accumulation  # = 1/K

pg_loss = agg_loss(pg_losses, response_mask, loss_agg_mode, **config.global_batch_info)
# ↑ 在 dp_actor 路径下，global_batch_info 为空 → 等价于 masked_mean

policy_loss = pg_loss  # (减去 entropy 等)
loss = policy_loss * loss_scale_factor             # ← 乘以 1/K = 除以 K
loss.backward()
```

### 2.3 对比

| 步骤 | v0.3.x | v0.7+ | 相同？ |
|------|--------|-------|--------|
| pg_loss 聚合 | `masked_mean(losses, mask)` | `agg_loss(...)` → 默认等价于 `masked_mean` | ✅ 相同 |
| 除以 K | `loss / gradient_accumulation` | `loss * (1/gradient_accumulation)` | ✅ 相同 |
| backward | `loss.backward()` | `loss.backward()` | ✅ 相同 |
| K 的公式 | `ppo_mini_batch_size // ppo_micro_batch_size_per_gpu` | 同左 | ✅ 相同 |

**结论：v0.3 和 v0.7 在 dp_actor 路径下的有效梯度完全相同。**

## 三、关于 `config.global_batch_info`

### 3.1 `compute_policy_loss_vanilla` 确实使用了 config

```python
# core_algos.py
pg_loss = agg_loss(
    loss_mat=pg_losses, loss_mask=response_mask,
    loss_agg_mode=loss_agg_mode,
    **config.global_batch_info   # ← 会解包 config 中的信息
)
```

### 3.2 dp_actor 路径下 `global_batch_info` 为空

`ActorConfig.global_batch_info` 的默认值是 `field(default_factory=dict)` → 空字典 `{}`。

在 dp_actor.py 路径中，没有代码填充这个字典。因此 `agg_loss(**{})` 使用默认参数：
- `dp_size = 1`
- `batch_num_tokens = None` → fallback 到 `loss_mask.sum()`（本地 tokens）

等效于：`pg_loss = masked_mean(pg_losses, response_mask)`，与 v0.3 完全一致。

### 3.3 Engine 路径（非 dp_actor）会填充 `global_batch_info`

当通过 `verl/workers/utils/losses.py` 的 `ppo_loss()` 函数调用时，
`global_batch_info` 会被填充（dp_size, batch_num_tokens, global_batch_size 等）。
但这个路径在 FSDP workers 中不被使用。

## 四、gradient_accumulation 的精确定义

### 4.1 K = ppo_mini_batch_size / ppo_micro_batch_size_per_gpu

```python
# dp_actor.py 第 562-563 行（v0.3 和 v0.7 都是这样）
self.gradient_accumulation = (
    self.config.ppo_mini_batch_size // self.config.ppo_micro_batch_size_per_gpu
)
```

### 4.2 K 不包含 dp_size

- **代码事实**：公式中不包含 dp_size
- **数据分发**：data 到达 dp_actor 前已被 dispatch 按 dp_size 切分
- **实际 micro-batch 数量**：每个 rank 的实际 micro-batch 数 = K / dp_size
  （因为每个 rank 只有 local_data / micro_batch_size 个 micro-batch）
- **但 loss_scale_factor 仍是 1/K**（不是 1/(K/dp_size)）

### 4.3 ppo_mini_batch_size 是全局值

**文件：`docs/algo/ppo.md`**
> `actor_rollout_ref.actor.ppo_mini_batch_size`: **The ppo_mini_batch_size is a global size across all workers**

## 五、Metric 报告格式的差异

虽然有效梯度相同，但 metric 报告格式在两个版本之间有差异：

### v0.3.x

```python
data = {'actor/pg_loss': pg_loss.detach().item(), ...}  # 原始 pg_loss
append_to_dict(metrics, data)  # → 每个 micro-batch 追加到 list
```

`metrics["actor/pg_loss"]` 是一个 **list**（每个 micro-batch 一个原始 pg_loss 值）。

### v0.7+

```python
metrics["actor/pg_loss"] += pg_loss.detach().item() * loss_scale_factor  # 乘以 1/K 后累加
```

`metrics["actor/pg_loss"]` 是一个 **标量**（所有 micro-batch 的缩放 pg_loss 之和）。

### 差异分析

假设有 K_actual = K/D 个 micro-batch，每个 pg_loss ≈ pg_avg：

| 版本 | 报告值 | 说明 |
|------|--------|------|
| v0.3 | list of K_actual values, each ≈ pg_avg | 下游处理可能取 mean 或 sum |
| v0.7 | K_actual × pg_avg / K ≈ pg_avg / D | 标量，已经缩放过 |

**如果下游处理对 v0.3 取平均，最终报告 = pg_avg。**
**v0.7 的最终报告 = pg_avg / D。**

这意味着在日志中看到的 `actor/pg_loss` 数值可能因为报告方式不同而有差异，
但这**不影响实际的训练梯度**。

## 六、v0.3 与 v0.7 的真正区别

既然 loss 缩放是相同的，那么两个版本之间真正的区别是什么？

### 6.1 算法层面

| 特性 | v0.3.x | v0.7+ |
|------|--------|-------|
| Clip 范围 | 单一 `cliprange` | 分离的 `clip_ratio_low`、`clip_ratio_high` |
| Dual-clip | 无 | 有 (`clip_ratio_c`，默认 3.0) |
| Log-ratio clamp | 无 | `torch.clamp(neg_approx_kl, -20.0, 20.0)` |
| Rollout IS correction | 无 | 支持 (`rollout_is_weights`) |
| Loss 聚合模式 | 仅 `masked_mean` | 多种 (`token-mean`, `seq-mean-*` 等) |

### 6.2 config 与代码结构

| 特性 | v0.3.x | v0.7+ |
|------|--------|-------|
| 函数签名 | 返回 `(loss, clipfrac, kl)` | 返回 `(loss, metrics_dict)` |
| Policy loss 注册 | 固定函数 | `@register_policy_loss` 插件系统 |
| Metric 报告 | `append_to_dict`（list） | `+=` 累加（scalar）+ `append_to_dict`（其他 metrics） |
| `global_batch_info` | 不存在 | 存在但在 dp_actor 路径下为空 |

### 6.3 不是区别的部分

| 项目 | 说明 |
|------|------|
| loss / gradient_accumulation | 两版**都做** |
| pg_loss 聚合方式（dp_actor 路径） | 两版都是 `masked_mean`（等价） |
| 有效梯度 | **完全相同** |

## 七、关于用户观测到的 ~12x 差异

用户观测到 4B（1 节点）和 32B（2 节点）都差 ~12 倍。

### 可能的原因

1. **K = 12 in the config**：如果 `ppo_mini_batch_size / ppo_micro_batch_size_per_gpu = 12`，
   那么 metric 报告格式的差异（v0.7 报告 scaled sum vs v0.3 报告 raw list）可以解释 ~12x 差异。

2. **两个节点都差 ~12x 说明差异与 dp_size 无关**：如果差异是 dp_size 相关的，
   1 节点（8 GPU）和 2 节点（16 GPU）应该有不同的倍数。两者相同 → 与 K（不含 dp_size）一致。

### 验证步骤

请确认你的配置：
```yaml
actor_rollout_ref:
  actor:
    ppo_mini_batch_size: ???   # 全局 mini-batch 大小
    ppo_micro_batch_size_per_gpu: ???  # 每卡 micro-batch 大小
```

计算 K = ppo_mini_batch_size / ppo_micro_batch_size_per_gpu。
如果 K ≈ 12，那么 metric 报告格式差异完全解释了你观测到的 ~12x。
