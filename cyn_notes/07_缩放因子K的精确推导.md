# 缩放因子 K 的精确推导与源码验证

> 本文档回应用户的两个质疑：
> 1. K = gradient_accumulation_steps 与实际观测不一致？
> 2. gradient_accumulation_steps 是否应该包含 dp_size（GPU 数量）？
>
> 结论：**K = ppo_mini_batch_size / ppo_micro_batch_size_per_gpu 是正确的，且与 dp_size 无关。**

---

## 一、源码验证：gradient_accumulation 的精确定义

### 1.1 代码位置

**文件：`verl/workers/actor/dp_actor.py`（第 562-563 行）**

```python
self.gradient_accumulation = (
    self.config.ppo_mini_batch_size // self.config.ppo_micro_batch_size_per_gpu
)
```

**结论：代码中 K = ppo_mini_batch_size / ppo_micro_batch_size_per_gpu，不涉及 dp_size。**

### 1.2 ppo_mini_batch_size 是全局的还是每卡的？

**文件：`docs/algo/ppo.md`**

```
- `actor_rollout_ref.actor.ppo_mini_batch_size`: The set of sampled trajectories
  is split into multiple mini-batches with batch_size=ppo_mini_batch_size for PPO
  actor updates. **The ppo_mini_batch_size is a global size across all workers**
```

**结论：ppo_mini_batch_size 是全局的（across all workers），不是每卡的。**

### 1.3 数据在到达 dp_actor 之前已经被 dp_size 切分

**文件：`verl/workers/fsdp_workers.py`（第 1024-1026 行）**

```python
@register(dispatch_mode=make_nd_compute_dataproto_dispatch_fn(mesh_name="actor"))
@DistProfiler.annotate(color="red", role="actor_update")
def update_actor(self, data: DataProto):
```

`make_nd_compute_dataproto_dispatch_fn` 会自动将数据按 dp_size 切分，
每个 DP rank 只收到 `global_batch / dp_size` 个样本。

**文件：`verl/single_controller/base/decorator.py`（第 70-87 行）**

```python
def _split_args_kwargs_data_proto(chunks, *args, **kwargs):
    """Split DataProto across chunks using DataProto.chunk()"""
    for arg in args:
        chunked_arg = BatchData(arg).chunk(chunks=chunks)
        # ...
```

### 1.4 关键数据流

```
Trainer 侧:
  rollout_batch (全局) ──────────────────────────────────────────┐
                                                                  │
  @register(dispatch_mode=make_nd_compute_dataproto_dispatch_fn)  │
      → DataProto.chunk(dp_size)                                 │
      → 每个 DP rank 收到 rollout_batch / dp_size 个样本         │
                                                                  │
dp_actor.py update_policy():                                      │
  data = 每个 rank 的本地数据 (= rollout_batch / dp_size)         │
  mini_batches = data.split(ppo_mini_batch_size)          ◄───────┘
  # ↑ ppo_mini_batch_size 是全局值
  # ↑ 如果 local_data < ppo_mini_batch_size，只有一个 mini_batch

  gradient_accumulation = ppo_mini_batch_size / ppo_micro_batch_size_per_gpu  # 全局 K
  micro_batches = mini_batch.split(ppo_micro_batch_size_per_gpu)
  # ↑ 实际 micro_batches 数量 = local_mini_batch / ppo_micro_batch_size_per_gpu

  for micro_batch in micro_batches:
      pg_loss = agg_loss(...)           # ≈ masked_mean_local（因为 global_batch_info 为空）
      loss = pg_loss * (1/K)            # loss_scale_factor = 1/K
      loss.backward()

  optimizer_step()                      # FSDP 自动做梯度 all-reduce 平均
```

## 二、每个 rank 的实际 micro-batch 数量

### 2.1 典型场景

设:
- `ppo_mini_batch_size = M`（全局）
- `ppo_micro_batch_size_per_gpu = m`
- `dp_size = D`
- `rollout_batch_size = N`（全局）

每个 rank 收到的样本数：`N / D`

如果只有一轮 PPO epoch 且 `N = M`（典型设置）：
- 每个 rank 的 mini_batch 大小 = `M / D`
- **实际 micro-batch 数** = `(M / D) / m = K / D`
- **gradient_accumulation (K)** = `M / m`

### 2.2 用户可能的配置

假设用户的配置使得 K = 12：

| 配置 | 可能的值 |
|------|---------|
| ppo_mini_batch_size = 48, ppo_micro_batch_size_per_gpu = 4 | K = 12 |
| ppo_mini_batch_size = 96, ppo_micro_batch_size_per_gpu = 8 | K = 12 |
| ppo_mini_batch_size = 192, ppo_micro_batch_size_per_gpu = 16 | K = 12 |

**用户观测到 4B（1 节点）和 32B（2 节点）都差 ~12 倍，这恰好说明 K 与 dp_size 无关！**

如果 K 依赖 dp_size（= num_gpus）：
- 1 节点 (8 GPU): K_alt = M / (8 * m)
- 2 节点 (16 GPU): K_alt = M / (16 * m)
- K_alt 不同 → 观测到的倍数应该不同 ← **矛盾！**

## 三、数学推导：为什么 K 与 dp_size 无关

### 3.1 v0.7+ 的有效梯度

每个 DP rank r 上有 `K/D` 个 micro-batch（K_per_rank = K/D）。

每个 micro-batch j 的梯度贡献（backward 输入）:
```
g_{r,j} = ∇(masked_mean_{r,j}) * (1/K)
```

rank r 的累积梯度:
```
G_r = Σ_{j=1}^{K/D} g_{r,j} = (K/D) * avg_∇_r / K = avg_∇_r / D
```

FSDP 梯度 all-reduce（取平均）:
```
G_effective = (1/D) * Σ_r G_r
            = (1/D) * Σ_r avg_∇_r / D
            = (1/D²) * Σ_r avg_∇_r
            ≈ (1/D²) * D * avg_∇_global
            = avg_∇_global / D
```

### 3.2 v0.3.x 的有效梯度

v0.3.x 没有 1/K 缩放：

每个 micro-batch j 的梯度贡献:
```
g_{r,j}^{old} = ∇(masked_mean_{r,j})
```

rank r 的累积梯度:
```
G_r^{old} = Σ_{j=1}^{K/D} g_{r,j}^{old} = (K/D) * avg_∇_r
```

FSDP 梯度 all-reduce:
```
G_effective^{old} = (1/D) * Σ_r (K/D) * avg_∇_r
                  = (K/D²) * Σ_r avg_∇_r
                  ≈ (K/D²) * D * avg_∇_global
                  = K * avg_∇_global / D
```

### 3.3 比值

```
G_effective^{old} / G_effective^{new}
= (K * avg_∇ / D) / (avg_∇ / D)
= K
```

**dp_size (D) 在分子分母中完全抵消！比值恒为 K。**

这就是为什么用户在 1 节点 (D=8) 和 2 节点 (D=16) 上观测到相同的 ~12 倍差异。

## 四、dp_actor.py 中 agg_loss 的实际行为

### 4.1 关键发现：global_batch_info 在 dp_actor.py 中为空

在 `verl/workers/actor/dp_actor.py` 中搜索 `global_batch_info`、`batch_num_tokens`、`dp_size`：

```
$ grep -n "global_batch_info\|batch_num_tokens\|dp_size" verl/workers/actor/dp_actor.py
（无匹配结果）
```

**`config.global_batch_info` 在 dp_actor.py 路径中从未被设置！**

这意味着 `compute_policy_loss_vanilla` 调用 `agg_loss(**config.global_batch_info)` 时，
传入的是空字典。`agg_loss` 使用默认参数：
- `dp_size = 1`
- `batch_num_tokens = None` → fallback 到 `loss_mask.sum()`（本地 micro-batch tokens）

### 4.2 等效行为

```python
# agg_loss 在 dp_actor.py 路径下的等效行为
batch_num_tokens = loss_mask.sum()  # 本地 micro-batch 的有效 token 数
loss = masked_sum(loss_mat, loss_mask) / batch_num_tokens * 1
     = masked_mean_local  # ← 与 v0.3.x 完全一致！
```

**所以 pg_loss 的值本身（在 scaling 之前）与 v0.3.x 相同。**

### 4.3 K 因子来自 loss_scale_factor

```python
# dp_actor.py line 585
loss_scale_factor = 1 / self.gradient_accumulation  # = 1/K

# dp_actor.py line 663
loss = policy_loss * loss_scale_factor              # = masked_mean / K
loss.backward()
```

**K 因子完全来自 `loss_scale_factor`，不来自 `agg_loss` 的全局归一化。**

### 4.4 对比 02 号文档的修正

02 号文档第 4 节说：
```
N_global = N_micro × K × D = 全局有效 token 总数（所有 DP workers、所有 micro-batches）
pg_loss_new = S_micro / N_global * D = S_micro / (N_micro * K)
```

**这对 dp_actor.py 路径是不准确的。** 更准确的描述是：

```
# dp_actor.py 路径（FSDP 用户）
pg_loss_new = S_micro / N_micro_local            # ← 与 v0.3.x 相同（masked_mean）
loss_backward = pg_loss_new / K                   # ← K 因子来自 loss_scale_factor

# ENGINE 路径（Megatron 用户）
batch_num_tokens = 全局 mini-batch tokens (all-reduced)
pg_loss_engine = S_micro / batch_num_tokens * dp_size  # ← K 因子在这里体现
loss_backward = pg_loss_engine                         # ← 无额外 scaling
```

两条路径的最终效果相同（有效梯度差 K 倍），但机制不同。

## 五、对用户的两个质疑的回答

### 质疑 1：K = gradient_accumulation_steps 与实际统计不一致？

**回答**：K = ppo_mini_batch_size / ppo_micro_batch_size_per_gpu 是正确的公式。

请检查你的实际配置：
```python
K = ppo_mini_batch_size / ppo_micro_batch_size_per_gpu
```

如果你的 K ≈ 12，那与你观测到的 ~12 倍差异一致。
如果你的 K ≠ 12，请检查：
1. 是否考虑了 `rollout.n` 的影响（trainer 代码中会乘以 `rollout.n`）
2. 是否使用了 `use_dynamic_bsz`（动态 batch 会改变 loss_scale_factor 的计算方式）

### 质疑 2：gradient_accumulation 应该是 ppo_mini_batch_size / (num_gpus × micro_batch)？

**回答**：不应该包含 dp_size。原因：

1. **代码事实**：`gradient_accumulation = ppo_mini_batch_size // ppo_micro_batch_size_per_gpu`，不包含 dp_size（第 562-563 行）

2. **数学推导**：dp_size 在新旧版本的梯度比值中完全抵消（见第三节推导）

3. **经验验证**：你自己观测到 1 节点 (8 GPU) 和 2 节点 (16 GPU) 的差异倍数相同（~12x），这恰好证明 K 与 dp_size 无关。如果 K 依赖 dp_size，两个设置的倍数应该不同。

4. **直觉解释**：虽然每次 forward 确实全局处理 `dp_size × micro_batch` 个样本，但 FSDP 的梯度平均（÷ dp_size）会在新旧版本中被同等应用。这个 dp_size 因子在比值中抵消了。

## 六、实际中影响 K 的额外因素

虽然 K = ppo_mini_batch_size / ppo_micro_batch_size_per_gpu 是精确公式，
但以下因素可能导致观测值略有偏差：

### 6.1 rollout.n 的影响

```python
# verl/trainer/ppo/ray_trainer.py line 1219-1220
ppo_mini_batch_size = self.config.actor_rollout_ref.actor.ppo_mini_batch_size
ppo_mini_batch_size = ppo_mini_batch_size * self.config.actor_rollout_ref.rollout.n
```

如果 `rollout.n > 1`，实际使用的 `ppo_mini_batch_size` 会被放大。
但这只影响 trainer 侧的数据量，dp_actor.py 中读取的仍是 config 原始值。

### 6.2 use_dynamic_bsz

```python
# dp_actor.py line 582-583
if self.config.use_dynamic_bsz:
    loss_scale_factor = response_mask.shape[0] / self.config.ppo_mini_batch_size
```

动态 batch 模式下，loss_scale_factor 不再是 1/K，而是实际 batch 大小与 ppo_mini_batch_size 的比值。

### 6.3 Token 数量的波动

由于不同样本的 response 长度不同，每个 micro-batch 的有效 token 数可能有较大波动。
这不影响 K 的公式，但会影响每个 micro-batch 的 `pg_loss`（masked_mean）的值。

## 七、总结

| 项目 | 结论 |
|------|------|
| K 的公式 | K = ppo_mini_batch_size / ppo_micro_batch_size_per_gpu ✅ |
| 是否包含 dp_size | **不包含** ✅ |
| ppo_mini_batch_size 的含义 | **全局**（across all workers），不是每卡 |
| K 因子在 dp_actor.py 的来源 | `loss_scale_factor = 1/K`（不是 agg_loss） |
| K 因子在 engine 路径的来源 | agg_loss 的 batch_num_tokens（覆盖整个 mini-batch） |
| 用户观测 ~12x | 与 K = 12 一致（请验证配置） |

### 验证步骤

请在你的训练配置中确认：
```yaml
actor_rollout_ref:
  actor:
    ppo_mini_batch_size: ???   # 全局 mini-batch 大小
    ppo_micro_batch_size_per_gpu: ???  # 每卡 micro-batch 大小
```

计算 K = ppo_mini_batch_size / ppo_micro_batch_size_per_gpu。如果 K ≈ 12，公式正确。
