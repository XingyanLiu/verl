# v0.3.x vs v0.7+ pg_loss 计算方式差异分析

## 1. v0.3.x 的 pg_loss 计算

> 源码位置：`verl/trainer/ppo/core_algos.py`（v0.3.0.post1 tag）
> GitHub 链接：https://github.com/volcengine/verl/blob/v0.3.0.post1/verl/trainer/ppo/core_algos.py

### 1.1 核心函数

```python
def compute_policy_loss(old_log_prob, log_prob, advantages, eos_mask, cliprange):
    negative_approx_kl = log_prob - old_log_prob
    ratio = torch.exp(negative_approx_kl)
    ppo_kl = verl_F.masked_mean(-negative_approx_kl, eos_mask)

    pg_losses = -advantages * ratio
    pg_losses2 = -advantages * torch.clamp(ratio, 1.0 - cliprange, 1.0 + cliprange)

    # ===== 关键：直接在 micro-batch 内做 masked_mean =====
    pg_loss = verl_F.masked_mean(torch.max(pg_losses, pg_losses2), eos_mask)

    pg_clipfrac = verl_F.masked_mean(torch.gt(pg_losses2, pg_losses).float(), eos_mask)
    return pg_loss, pg_clipfrac, ppo_kl
```

### 1.2 masked_mean 的实现

```python
# verl/utils/torch_functional.py
def masked_mean(values, mask, axis=None):
    """对 mask 为 1 的位置求平均"""
    s = masked_sum(values, mask, axis)
    return s / (mask.sum(axis=axis) + 1e-8)
```

### 1.3 v0.3.x 特点

- **只有一个 cliprange**：没有 `clip_ratio_low` / `clip_ratio_high` 分离
- **没有 dual-clip**：没有 `clip_ratio_c` 下界裁剪
- **归一化方式**：`masked_mean` 在 micro-batch 内的有效 token 上求平均
- **分母**：`eos_mask.sum()` = 当前 micro-batch 的有效 token 数（例如 512 tokens）

### 1.4 等效公式

```
pg_loss = Σ(loss_i * mask_i) / Σ(mask_i)    # i 遍历 micro-batch 内所有 token
```

## 2. v0.7+（当前版本）的 pg_loss 计算

> 源码位置：`verl/trainer/ppo/core_algos.py`（当前版本）
> 代码行号参考：
> - `compute_policy_loss_vanilla`: lines 1278-1369
> - `agg_loss`: lines 1138-1199

### 2.1 核心函数

```python
@register_policy_loss("vanilla")
def compute_policy_loss_vanilla(
    old_log_prob, log_prob, advantages, response_mask,
    loss_agg_mode="token-mean", config=None, rollout_is_weights=None,
):
    clip_ratio = config.clip_ratio
    clip_ratio_low = config.clip_ratio_low if config.clip_ratio_low is not None else clip_ratio
    clip_ratio_high = config.clip_ratio_high if config.clip_ratio_high is not None else clip_ratio
    clip_ratio_c = config.get("clip_ratio_c", 3.0)  # dual-clip 下界

    negative_approx_kl = log_prob - old_log_prob
    negative_approx_kl = torch.clamp(negative_approx_kl, min=-20.0, max=20.0)  # 新增稳定性 clamp
    ratio = torch.exp(negative_approx_kl)

    pg_losses1 = -advantages * ratio
    pg_losses2 = -advantages * torch.clamp(ratio, 1 - cliprange_low, 1 + cliprange_high)
    clip_pg_losses1 = torch.maximum(pg_losses1, pg_losses2)

    # Dual-clip 处理 (DAPO 特性)
    pg_losses3 = -advantages * clip_ratio_c
    clip_pg_losses2 = torch.min(pg_losses3, clip_pg_losses1)
    pg_losses = torch.where(advantages < 0, clip_pg_losses2, clip_pg_losses1)

    # ===== 关键变更：使用 agg_loss 进行全局归一化 =====
    pg_loss = agg_loss(
        loss_mat=pg_losses,
        loss_mask=response_mask,
        loss_agg_mode=loss_agg_mode,
        **config.global_batch_info  # 包含 batch_num_tokens, dp_size, global_batch_size
    )
    return pg_loss, pg_metrics
```

### 2.2 agg_loss 函数（关键变更点）

```python
def agg_loss(loss_mat, loss_mask, loss_agg_mode, dp_size=1,
             batch_num_tokens=None, global_batch_size=None, loss_scale_factor=None):
    if loss_agg_mode == "token-mean":
        if batch_num_tokens is None:
            if dp_size > 1:
                raise ValueError("(global) batch_num_tokens is required when dp_size > 1")
            batch_num_tokens = loss_mask.sum()  # fallback: 仅当 dp_size=1 时
        # ===== 关键：用 GLOBAL batch_num_tokens 归一化 =====
        loss = verl_F.masked_sum(loss_mat, loss_mask) / batch_num_tokens * dp_size
    # ...
```

### 2.3 batch_num_tokens 的来源

`batch_num_tokens` 通过 AllReduce 在所有 DP workers 上聚合：

```python
# verl/workers/engine/fsdp/transformer_impl.py (line ~579)
batch_num_tokens = data["loss_mask"].sum().to(get_device_id())
torch.distributed.all_reduce(
    batch_num_tokens, op=torch.distributed.ReduceOp.SUM, group=self.get_data_parallel_group()
)
```

然后通过 `global_batch_info` 传递到 `agg_loss`:
```python
# verl/workers/utils/losses.py (line ~65)
config.global_batch_info["dp_size"] = data["dp_size"]
config.global_batch_info["batch_num_tokens"] = data["batch_num_tokens"]
```

### 2.4 v0.7+ 等效公式 (token-mean 模式)

```
pg_loss_micro = Σ(loss_i * mask_i) / batch_num_tokens_global * dp_size
                                     ^^^^^^^^^^^^^^^^^^^^^^^^
                                     全局有效 token 总数（所有 DP workers、所有 micro-batches）
```

## 3. 关键差异对比

### 3.1 归一化分母

| 版本 | 分母 | 典型值举例 |
|------|------|-----------|
| v0.3.x | `eos_mask.sum()` (micro-batch内) | ~512 tokens |
| v0.7+ | `batch_num_tokens` (全局) | ~32768 tokens |

假设 micro-batch 大小 = 512 tokens，global batch = 32768 tokens（8 GPU × 8 accumulation steps × 512 tokens/micro-batch），则：

- v0.3.x: `pg_loss ≈ loss_sum / 512`
- v0.7+: `pg_loss ≈ loss_sum / 32768 * 8 = loss_sum / 4096`

**pg_loss 缩小到原来的 512/4096 = 1/8**（在这个例子中，即变为原来的 12.5%）

### 3.2 梯度行为差异

虽然报告的 loss 值不同，但有效梯度是否相同取决于梯度累积策略：

**v0.3.x**: 每个 micro-batch 计算 loss → backward → 梯度累加 → 除以 accumulation steps → optimizer.step()
- 有效梯度 ∝ `Σ_i (sum_i / N_i_tokens) / n_accum`

**v0.7+**: 每个 micro-batch 计算 loss（已包含全局归一化）→ backward → 梯度累加 → optimizer.step()
- 有效梯度 ∝ `Σ_i (sum_i / N_global_tokens * dp_size)`
- = `total_sum / N_global_tokens * dp_size`

两者的有效梯度**在数学上等价**（只是 scale 不同），但如果学习率是针对旧版本调优的，迁移后需要调整。

### 3.3 除归一化外的其他差异

| 特性 | v0.3.x | v0.7+ |
|------|--------|-------|
| Clip 范围 | 单一 `cliprange` | 分离的 `clip_ratio_low` 和 `clip_ratio_high` |
| Dual-clip | 无 | 有 (`clip_ratio_c`，默认 3.0) |
| Log-ratio clamp | 无 | 有 (`torch.clamp(negative_approx_kl, -20.0, 20.0)`) |
| Rollout IS correction | 无 | 支持 (`rollout_is_weights`) |
| 函数签名 | 返回 `(loss, clipfrac, kl)` 元组 | 返回 `(loss, metrics_dict)` |
| Loss aggregation | 仅 `masked_mean` | 多种模式 (`token-mean`, `seq-mean-token-sum`, 等) |

### 3.4 DAPO 文档中的代码片段已过时

注意 `docs/algo/dapo.md` 中的代码片段：

```python

    loss = verl_F.masked_mean(loss_mat, loss_mask)   # <-- 旧式 micro-batch 归一化
```

但实际代码 (`core_algos.py` line 1168-1173) 是：


**文档与实际代码不一致！** 这可能进一步加重了用户的困惑。

## 4. 量级差异的数学推导

> **⚠️ 修正说明（见 `07_缩放因子K的精确推导.md`）**：
>
> 下面的推导针对的是 **ENGINE 路径**（Megatron 或 engine-based actor），其中 `agg_loss` 使用
> 全局 `batch_num_tokens`（覆盖整个 mini-batch）。
>
> **对于 FSDP 用户（dp_actor.py 路径）**，`config.global_batch_info` 为空，`agg_loss` 使用的是
> 本地 micro-batch 的 token 数。因此 `pg_loss` 的值本身与 v0.3.x 相同（masked_mean_local），
> K 因子来自 `loss_scale_factor = 1/K`（dp_actor.py 第 585 行），而非 agg_loss 的分母。
>
> 两条路径的最终效果相同：有效梯度差 K 倍。
>
> **K = ppo_mini_batch_size / ppo_micro_batch_size_per_gpu，与 dp_size 无关。**

设：
- `N_micro` = 单个 micro-batch 的有效 token 数
- `K` = ppo_mini_batch_size / ppo_micro_batch_size_per_gpu（**与 dp_size 无关**）
- `D` = data parallel size (dp_size)
- `N_global = N_micro × K × D` = 全局有效 token 总数（仅 ENGINE 路径使用）
- `S_micro` = 单个 micro-batch 的 loss 求和值

### v0.3.x 报告的 pg_loss (每个 micro-batch)
```
pg_loss_old = S_micro / N_micro
```

### v0.7+ (ENGINE 路径) 报告的 pg_loss (每个 micro-batch)
```
pg_loss_engine = S_micro / N_global * D = S_micro / (N_micro * K * D) * D = S_micro / (N_micro * K)
```

### v0.7+ (dp_actor.py 路径) 的 backward loss (每个 micro-batch)
```
pg_loss_dp = S_micro / N_micro                  # 与 v0.3.x 相同
loss_backward = pg_loss_dp * (1/K) = S_micro / (N_micro * K)   # loss_scale_factor 缩放
```

### 倍数关系
```
pg_loss_new / pg_loss_old = 1 / K   (ENGINE 路径：agg_loss 分母扩大)
loss_backward / pg_loss_old = 1 / K  (dp_actor 路径：loss_scale_factor 缩放)
```

**结论：有效梯度缩小的倍数等于 K = ppo_mini_batch_size / ppo_micro_batch_size_per_gpu。**

K 与 dp_size 无关（FSDP 梯度平均在新旧版本中同等应用，在比值中抵消）。

（注意：这个推导假设所有 micro-batch 的 token 数大致相等。实际中由于 padding 和 mask 差异，可能还会有额外的波动。）
