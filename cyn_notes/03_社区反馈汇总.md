# 社区反馈汇总

## 1. GitHub Issues

### Issue #2776: dapo token mean loss
> https://github.com/verl-project/verl/issues/2776

**问题**：社区用户指出 DAPO 论文中 token mean loss 应在 **group 内** 计算（即同一 prompt 的多条 response 组成的 group），但 verl 代码中是在 **batch 内** 做 token-mean。这导致了归一化维度不一致。

**关键讨论内容**：
- DAPO 论文的理论意图：token mean 应该在每个 prompt group 内部计算，然后再对 groups 求平均
- verl 实现：response groups 会被 shuffle 后分发到不同 GPU，所以严格的 group-level 归一化在分布式训练中不易实现
- 折中方案：设置 `micro_batch_size_per_gpu=1` 时，每个 micro-batch 只有一个序列，此时 batch-level mean ≈ sequence-level mean
- **结论**：当前实现是出于分布式效率的权衡，但用户需知晓这个差异

### Issue #957: loss_agg_mode in the recipe of DAPO
> https://github.com/verl-project/verl/issues/957

**问题**：用户指出 DAPO 的三种 loss 聚合模式容易混淆：
- `seq-mean-token-mean`: DAPO 使用的方式（论文原始意图）
- `seq-mean-token-sum`: DrGRPO 使用的方式
- `token-mean`: 原始 GRPO 使用的方式

但实际 DAPO recipe 脚本 (`run_dapo_early_qwen2.5_32b.sh`) 用了 `seq-mean-token-sum`，用户怀疑这是因为早期版本 `seq-mean-token-mean` 与 `seq-mean-token-sum` 的代码实现写成了一样的（bug），导致没发现问题。

**结论**：不同版本的默认配置和实现可能不一致，升级后一定要核查 `loss_agg_mode` 的实际行为。

---

### Issue #593: about grpo loss computation
> https://github.com/verl-project/verl/issues/593

**问题**：社区开发者指出 GRPO 论文要求先在每条序列的 token 维度上求平均，再对序列求平均（`seq-mean-token-mean`），但 verl 实现直接做了全局 `masked_mean`（`token-mean`）。并用数值示例证明两者**不等价**：

```python
batch = [[2, 2, 2, 2, 2, ...], [10, 3, 4, 5, 2, ...]]
eos_mask = [[1, 1, 1, 1, 1, 0, ...], [1, 0, 0, 0, 0, 0, ...]]

# token-mean (全局)
masked_mean(batch, eos_mask) = 3.3333

# seq-mean-token-mean (先序列内平均再求平均)
masked_mean(batch, eos_mask, axis=1).mean() = 6.0
```

**差异高达 ~2 倍**，且序列长度差异越大，偏差越大。

**结论**：这个 issue 直接推动了 `loss_agg_mode` 参数的引入。

### Issue #3129: Do these Metrics really need to be scaled?
> https://github.com/volcengine/verl/issues/3129

**问题**：用户质疑 loss metrics（如 `pg_loss`、`kl_loss`）报告时是否需要乘以 `loss_scale_factor`。

**讨论**：
- 对于 **backward pass**：scaling 是必要的，确保 loss 值"并行度无关"
- 对于 **metric reporting**：scaling 可能是多余或误导的，因为 metric 通常用 mean reduction

**结论**：维护者承认当前 metric 报告可能存在不一致，计划修复。这可能导致用户看到的 `pg_loss` 数值与实际优化效果之间有偏差。

---

### Issue #4852: bypass_mode 中 response_mask 修改导致 batch_num_tokens 不一致
> https://github.com/verl-project/verl/issues/4852

**问题**：在开启 bypass mode（rollout correction）时，`compute_policy_loss_bypass_mode` 函数内部会修正 `response_mask`，导致实际参与 `agg_loss` 的有效 token 数与训练前收集的 `batch_num_tokens` 不一致。

**影响**：在 `token-mean` 模式下，分母（`batch_num_tokens`）偏大，使得 pg_loss 进一步变小。

**代码位置**：
- mask 收集：`verl/workers/engine/megatron/transformer_impl.py` line ~476
- mask 修正：`verl/trainer/ppo/core_algos.py` line ~2065

---

### Issue #5625: Incorrect loss_scale_factor for token-mean mode in gradient accumulation
> https://github.com/verl-project/verl/issues/5625

**问题**：在 legacy FSDP (dp_actor.py) 中使用梯度累积时，`loss_scale_factor` 使用的是 sample-count ratio，但 `agg_loss` 对每个 micro-batch 用的是 local token count 归一化，两者不匹配。

**具体差异**：
- 当前累积的 loss: `L_accumulated = Σ_i (sum_i / tokens_i) × (samples_i / total_samples)`
- 正确的全 batch loss: `L_full = (Σ_i sum_i) / total_tokens`

**影响**：梯度累积下的 loss 与真正的全 batch loss 不等价，可能导致训练效果偏差。

## 3. 社区共识总结

1. **问题确实广泛存在**：多人在 verl 社区报告了类似的 pg_loss 量级变化问题
2. **根因明确**：`masked_mean`（micro-batch 归一化）→ `agg_loss`（global batch 归一化）的设计变更
3. **设计合理但缺乏迁移指导**：这个变更的目的是好的（并行度无关），但缺少升级迁移文档
4. **文档代码不一致**：DAPO 文档中的代码片段仍展示旧式的 `masked_mean`，但实际代码已使用 `agg_loss`
5. **多个衍生 bug**：bypass mode mask 不一致、loss_scale_factor 在梯度累积中的问题等
